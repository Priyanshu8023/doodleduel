# API.md — Socket & HTTP API Reference

Scribble is primarily a WebSocket-based application. This document covers both the Socket.IO event API and the HTTP API.

---

## HTTP API

### `GET /api/custom-health-check`

**Purpose**: Liveness probe for load balancers, container orchestration (Kubernetes readiness probe), and uptime monitors.

**Response**:
```json
{ "status": "ok" }
```

**Status codes**: `200 OK` (always, as long as the Node.js process is alive).

**Note**: This does not check Redis or PostgreSQL connectivity — it's a shallow process-alive check only. A production deployment should add a deep health check that verifies Redis `PING` and Postgres connectivity.

---

### `GET /*` (Next.js handler)

All other HTTP requests (page loads, static assets, API routes) are forwarded to Next.js's built-in request handler via `expressApp.use(handle)`. This handles:
- SSR / RSC rendering for `app/` routes.
- Static asset serving from `.next/static/` with cache-control headers.
- Next.js Image Optimization API (`/_next/image`).

---

## Socket.IO Event API

All events travel over a single persistent WebSocket connection. Socket.IO handles framing, acknowledgements, and room fan-out.

### Client → Server Events

---

#### `join_room`

**Payload**:
```typescript
{
  roomId: string;    // max 50 chars; validated server-side
  name: string;      // max 20 chars; validated server-side
  playerId: string;  // UUID generated by client (localStorage persisted)
}
```

**Validation**:
```typescript
if (!roomId || roomId.length > 50) return;
if (!name || name.length > 20) return;
```

**Side Effects**:
1. Socket joins the Socket.IO room `roomId`.
2. Room is created in Redis if it doesn't exist.
3. Player is added to the room's list/set (idempotent — duplicate check via `SISMEMBER`).
4. Socket ↔ Player mappings are updated (allows reconnection).
5. `room_updated` is broadcast to all sockets in the room.
6. `player_joined` is sent to all *other* sockets in the room (triggers canvas sync).

**Error handling**: Silent `return` on validation failure. No error event is emitted to the client.

---

#### `chat_message`

**Payload**:
```typescript
{
  roomId: string;
  message: string;  // max 100 chars (server) / max 50 chars (client UI)
}
```

**Validation**:
```typescript
if (!roomId || !message || message.length > 100) return;
```

**Flow**:
1. Look up `playerId` from `socket.id`.
2. Fetch player data from Redis.
3. Call `checkGuess()` — compares `message.toLowerCase()` to `currentWord.toLowerCase()`.
4. **If correct guess**: emit `system_message` (`CORRECT_GUESS`) + `room_state_updated` to room.
5. **If incorrect**: broadcast `receive_message` to room (chat message visible to all).

**Note**: The guess check is case-insensitive. A correct guess is never echoed as a chat message — this prevents revealing the word via the chat history.

---

#### `start_game`

**Payload**: `roomId: string`

**Authorization**: Server validates:
- Room must be in `LOBBY` status.
- Room must have ≥ 2 players total.
- Room must have ≥ 2 **active** (socket-connected) players.

There is no explicit host-only check on the server. The client UI only renders the "Start Game" button for `players[0]` (the first player who joined), but a client sending `start_game` out of turn would succeed if conditions are met.

---

#### `draw_batch`

**Payload**:
```typescript
{
  roomId: string;
  points: Array<{
    x: number;       // normalised [0, 1]
    y: number;
    prevX: number;
    prevY: number;
    color: string;   // CSS hex color, e.g. "#EF130B"
    size: number;    // normalised brush size
  }>;
}
```

**Validation**:
```typescript
if (!Array.isArray(points) || points.length > 200) return;
// Verifies: room status === "PLAYING" AND socket.playerId === room.drawerId
```

**Authorization check**: Only the current drawer's events are relayed. Any other player sending `draw_batch` is silently dropped — this is the primary anti-cheat mechanism for drawing.

---

#### `sync_canvas`

**Payload**:
```typescript
{
  targetSocketId: string;  // socket ID of newly joined player
  canvasData: string;      // base64 data URL: "data:image/png;base64,..."
}
```

**Note**: The server does NOT verify that the sender is the drawer. This is a trust vector — a malicious client could send arbitrary canvas data to any socket ID in the room. Since Socket.IO rooms are isolated (a socket in room A can't guess a socket ID in room B), the attack surface is limited to malicious players within the same room.

**Flow**: Server forwards `canvasData` directly to `targetSocketId` via `io.to(targetSocketId).emit("receive_canvas_sync", canvasData)`.

---

### Server → Client Events

---

#### `room_updated`

Broadcast to all sockets in a room whenever the authoritative room state changes.

**Payload** (full `RoomState`):
```typescript
{
  roomId: string;
  players: Array<{
    id: string;
    name: string;
    score: number;
    hasGuessed: boolean;
  }>;
  status: "LOBBY" | "CHOOSING_WORD" | "PLAYING" | "ROUND_END" | "GAME_OVER";
  drawerId: string | null;
  currentWord: string | null;  // Full word — clients MUST mask this if player is not drawer
  roundEndTime: number;        // Unix timestamp ms
  round: number;
  maxRounds: number;
}
```

> **Security note**: `currentWord` is sent in plaintext to all clients. A client with DevTools open can see the word. This matches skribbl.io behavior — the game relies on social trust. A hardened version would send the masked word to guessers and only the real word to the drawer.

---

#### `room_state_updated`

Alias of `room_updated`, emitted specifically after a correct guess (to differentiate real-time state refresh from full room transitions in client logic).

---

#### `receive_message`

Broadcast to all sockets in room when an **incorrect** guess or normal chat message is sent.

**Payload**:
```typescript
{
  userId: string;    // socket.id of sender
  userName: string;
  message: string;
}
```

---

#### `system_message`

Broadcast for significant game events.

**Payload variants**:
```typescript
// Correct guess
{ type: "CORRECT_GUESS", userId: string, userName: string }

// Round ended (word reveal)
{ type: "ROUND_END", word: string }
```

---

#### `player_joined`

Sent to all **existing** sockets in a room when a new player joins. Used to trigger drawer-side canvas sync.

**Payload**:
```typescript
{ newPlayerSocketId: string }
```

---

#### `draw_batch`

**Direction**: Server → All non-sender sockets in room.

Same payload as the client→server version. Server relays without mutation (except dropping unauthorized senders).

---

#### `receive_canvas_sync`

**Direction**: Server → single target socket only.

**Payload**: `canvasData: string` (base64 PNG data URL)

---

## Rate Limiting & Abuse Prevention

| Vector | Current Guard | Recommended Enhancement |
|---|---|---|
| Message flood (chat) | `message.length > 100` server, `> 50` client UI | Token bucket: max 5 messages/3 seconds per socket |
| Draw event flood | `points.length > 200` guard | Redis rate limit: max 100 draw_batch events/minute per socket |
| Room ID enumeration | No room listing API; rooms are join-by-link | Add UUID-style room IDs (current: user-defined `roomId`) |
| ReDoS / injection | No regex on chat; direct string comparison | Safe as-is |
| Canvas data poisoning | No validation on canvasData size | Validate `canvasData.length < 2MB` before forwarding |

---

## Error Handling Strategy

The current strategy is **fail-silently**: invalid or unauthorized requests return with no response. This is intentional for a game — emitting error events to clients for common invalid inputs (e.g., sending `draw_batch` when not the drawer) would add noisy event-handling on the client with no UX benefit.

For infrastructure errors (Redis unavailable), the `workerLoop` has a top-level `try/catch` that logs to stderr and continues. Socket event handlers do not have per-handler try/catch — a Redis error during event processing will throw an unhandled promise rejection (potential process crash risk in production). This should be addressed with a global `process.on('unhandledRejection', ...)` handler.
